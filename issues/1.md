### Issue Title
Column Classification + Routing (Stages 3–6, Target: 90%+ Accuracy)

**Track:** Ingestion Pipeline  
**Sprint:** S5–S6

---

### Tasks
- [ ] Map source columns to canonical schema using matched template
- [ ] Non-template columns enter classification pipeline
- [ ] Zero row drops — rows that cannot be parsed are extracted in raw form
- [ ] Confidence scoring and state assignment: Accepted, Provisional, Unresolved
- [ ] Routing rule: everything below Accepted threshold → Provisional (no direct path to Unresolved at launch)
- [ ] Implement Provisional tiering at routing time:
  - **Provisional-Reviewable:** user action required, system cannot resolve
  - **Provisional-Pending:** system can likely resolve when a specific registry event occurs — flagged at routing time
  - **Provisional-Pattern:** belongs to a group of 50+ identical items resolvable with one action — held for S7 exception pipeline
- [ ] Only Provisional-Reviewable items surface in the review queue

### Acceptance Criteria
- ≥90% of columns classified accurately
- Provisional tiering applied at routing time
- Review queue contains only Provisional-Reviewable items

### ⚠️ Failure Risk: Provisional Is a Roach Motel (Failure #5 — Severe)
A single large distributor file can generate thousands of Provisional rows. Without tiering, the review queue grows faster than any human can clear it. By S7 users stop engaging, coverage metrics degrade, and the anchor client loses confidence. The retry/replay system (S9–S10) is too late — items entered Provisional in S4 sit for 5+ sprints.
**Solution built in:** Provisional tiering separates actionable items from deferrable ones at routing time. Queue stays manageable. Pattern items are pre-grouped for the S7 exception pipeline. Pending items are pre-enrolled for retry.

**Milestone:** End of S6 — 100 Formats / Anchor Pilot Live
